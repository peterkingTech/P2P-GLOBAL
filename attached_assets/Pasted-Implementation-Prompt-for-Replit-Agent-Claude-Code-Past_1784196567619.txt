Phase 4 — Admin Translation Dashboard

4.1 Gap to close first: there is no persistent job/queue state

Right now translation happens via direct API calls (POST /translations/admin/trigger, POST /translations/admin/batch-curriculum with streamed progress) with no row-level record of attempts, failures, or history once the stream ends. The dashboard needs something to read. Add:

sql-- db/migrations/004_translation_jobs.sql
create table if not exists p2p_translation_jobs (
  id uuid primary key default gen_random_uuid(),
  content_type text not null,        -- 'lesson','module','curriculum','section','reflection_question', etc.
  content_id uuid not null,
  language text not null references p2p_languages(code),
  status text not null default 'pending',  -- pending, processing, completed, failed, retrying
  attempts int not null default 0,
  max_attempts int not null default 3,
  last_error text,
  triggered_by text not null default 'admin', -- 'admin' | 'batch' | user_id
  ai_provider text default 'gpt-4o-mini',
  ai_cost_usd numeric(10,4),
  created_at timestamptz not null default now(),
  started_at timestamptz,
  completed_at timestamptz
);
create index on p2p_translation_jobs (status, created_at);
create index on p2p_translation_jobs (content_type, content_id, language);

Update translationEngine.ts so every call (single trigger, batch curriculum, or automatic re-translation) writes a row to p2p_translation_jobs at start (processing) and updates it at the end (completed or failed with last_error, incrementing attempts). This is a logging/tracking layer over the existing engine — it does not change how translation itself works.

If ai_cost_usd isn't already tracked per call from the OpenAI response usage data, add it here (OpenAI returns token usage; compute cost from the gpt-4o-mini per-token rate and store it).
